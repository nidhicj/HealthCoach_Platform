## 1. What this product is

**Parivarthan** — webapp for independent health coaches (HCs) in India.

HCs use it to:
1. Onboard and manage clients
2. Run sessions with AI-assisted MOM (Minutes of Meeting) generation
3. Send action items to clients
4. Run between-session check-ins
5. Build a personalised AI writing style via snippet capture (HC edits → style library)

**Current user**: coach only. Client data model + auth exist in DB; no client UI built yet.  
**Geography**: India-first. DPDP Act 2023 applies.  
**Stage**: production repo. Prototype learnings only — no prototype code copied here.  
**Pilot**: 1 HC, ~5 clients. Not yet launched.

## 7. Discrepancies between docs and implementation

These are explicit mismatches between what a document says and what the code actually does. Read before making decisions that depend on either.

### D1 — SPEC-0001 says `is_first_session=true` for M000; code uses `session_number == 0`

**Where**: `SPEC-0001-hc-core-cycle.md` §Stage 2, paragraph 2: "sessions row created with `is_first_session=true`"  
**Reality**: There is no `is_first_session` column. M000 is detected as `session.session_number == 0`. This was explicitly decided in SESSION_LOG 2026-05-05 ("M000 detection: `session.session_number == 0` (not a non-existent `is_first_session` column)").  
**Impact**: SPEC-0001 Stage 2 acceptance criterion "M000 session creates `sessions` row with `is_first_session=true`" is untestable as written.  
**Action needed**: Update SPEC-0001 §Stage 2 to say `session_number == 0` instead of `is_first_session=true`.

### D2 — SPEC-0001 §Stage 2 says M000 MOM flow ends with a separate `/mom/send` endpoint

**Where**: SPEC-0001 §API surface shows `POST /api/sessions/{id}/mom/send`  
**Reality**: MOM send is done via `PATCH /api/sessions/{id}/mom` with `status='sent'` in the body. There is no separate `/mom/send` endpoint.  
**Impact**: If someone builds client-facing tooling from the SPEC-0001 API surface, they'll call the wrong endpoint.  
**Action needed**: Update SPEC-0001 §API surface to remove `/mom/send` and document that `PATCH /sessions/{id}/mom` handles the send transition.

### D3 — REPO-INDEX.md lists ADR-0003 as "Proposed (flip to Accepted before code)"

**Where**: `docs/REPO-INDEX.md` ADR table  
**Reality**: ADR-0003 itself says "Status: Accepted" (verified by reading the file). SESSION_LOG 2026-05-01 says "ADR-0003 flipped to Accepted before P1 coding started."  
**Impact**: REPO-INDEX misleads anyone using it to check ADR statuses.  
**Action needed**: Update REPO-INDEX.md ADR-0003 row to "Accepted".

### D4 — SPEC-0002-llm-service.md referenced in REPO-INDEX but does not exist

**Where**: `docs/REPO-INDEX.md` specs table: "Unit_001_HcCoreCycle/SPEC-0002-llm-service.md — Draft"  
**Reality**: The file is not in `docs/specs/Unit_001_HcCoreCycle/`. SESSION_LOG 2026-05-04 says "Moved `0004-llm-service.md` → `Unit_001_HcCoreCycle/SPEC-0002-llm-service.md`" but this didn't make it into the repo (possibly only done in claude.ai session, not committed via Claude Code).  
**Impact**: Reference to a non-existent spec.  
**Action needed**: Either create `SPEC-0002-llm-service.md` (content is the old LLM service spec), or remove the REPO-INDEX reference and confirm LLM service is fully captured by SPEC-0001 + ADR-0003.

### D5 — ADR-0001 says Next.js 15; code runs on Next.js 16.2.4

**Where**: ADR-0001 Decision table "Frontend framework: Next.js 15 (App Router, TypeScript)"; also echoed in PHASE-06-frontend.md §ADRs implemented.  
**Reality**: `frontend/package.json` and HANDOVER-P6 both confirm Next.js 16.2.4.  
**Impact**: Minor — intent unchanged (App Router, Tailwind, shadcn). But version-sensitive tooling checks would fail.  
**Action needed**: Add a changelog entry to ADR-0001 noting the version bump with the reason ("released between phases; intent unchanged").

### D6 — ADR-0001 and ADR-0003 document 4-model LLM chain; live chain has 3 models

**Where**: ADR-0001 §LLM model chain table lists 4 models including `openai/gpt-oss-120b:free`.  
**Reality**: SESSION_LOG 2026-05-04: "`openai/gpt-oss-120b:free` removed — not a valid OpenRouter slug. Chain is now llama-3.3-70b → gemma-3-27b → nemotron-3-super-120b-a12b." `backend/llm_config.yaml` is authoritative.  
**Impact**: Anyone reading ADR-0001 to understand the model chain will have the wrong picture.  
**Action needed**: ADR-0001 changelog entry: drop `gpt-oss-120b:free` from the chain table, note reason.

**Where**: ADR-0004 shows `prompts/` at repo root and `backend/src/llm/`  
**Reality**: Prompts live at `backend/prompts/`, not repo root. The LLM module is at `backend/src/llm_service/`, not `backend/src/llm/`.  
**Impact**: Layout mismatch could confuse anyone using ADR-0004 to navigate the repo.
